## add-agent-runtime — Grill残課題 (20260703)

### Q1. hidden_facts に per-fact の可視性(known_by/visibility)構造が定義されていない
- **対象**: `add-agent-runtime` design.md D1(`CharacterAgentInput.visible_facts`)、spec.md「Character Agent コンテキストのスコープ限定」要件(「本人が見てよい hidden_facts」)。参照先: `add-state-model` specs/state-model/spec.md「Scene モデル」要件(50行目)・「Visibility enum」要件(74-75行目)。
- **なぜ重要**: spec-foundation §4.3(1) は Character Agent コンテキストに「本人が見てよい hidden_facts」のみを含めよと定めるが、これは scene の各 hidden_fact に「誰が見てよいか」を表す per-item のメタデータ(known_by や visibility)が存在することを前提にする。しかし state-model の Scene モデル要件は `hidden_facts` を単に「reader_visible_facts と別のリスト」とだけ規定し、Visibility enum を持つ型として明示されているのは Event・canon/gm_vault 等の「状態エントリ」・StateDiffChange のみで、Scene.hidden_facts は含まれていない。つまり hidden_facts が単純な `list[str]` である可能性があり、その場合「本人が見てよい hidden_facts」は構造的に判定不能で、Context Builder の実装がこの不変条件を満たせない(全員に見せるか誰にも見せないかの二択になる)。
- **自己調査**: `add-state-model/specs/state-model/spec.md` の Scene モデル要件・Visibility enum 要件を確認したが、hidden_facts の要素型が visibility を持つかどうかは明記されていない。spec-foundation §5(データモデル概要)も「reader_visible_facts/hidden_facts」としか書いておらず、要素の構造までは規定していない。`add-state-model` は本 change の依存元であり、その capability spec を編集する権限は本 change 側にない。
- **検討した選択肢**: A) hidden_facts の要素を `Fact`(text + visibility + known_by)型に拡張するよう `add-state-model` 側に変更を依頼する / B) 本 change 側で「hidden_facts は現時点では per-character 可視性を持たないため、Character Agent コンテキストには一切含めない(reader_visible_facts のみを含める)」という保守的な解釈で実装し、spec の「本人が見てよい hidden_facts」文言をそちらに合わせて弱める / C) hidden_facts 自体に character id リストとの並列マッピング(別ファイルまたは同一 YAML 内の補助フィールド)を持たせる案を state-model 側に提案する。
- **推奨案**: B(保守的解釈)を暫定実装とし、A or C を `add-state-model` への追補 change として提案するのが安全側に倒す形で望ましい。情報スコープ違反(過剰包含)より過小包含の方がプロダクトの核契約(spec-foundation §4)に対するリスクが小さいため。
- **不足インプット**: `add-state-model` の hidden_facts 実データ構造(per-fact visibility を持たせる意図があったか)をユーザーまたは state-model spec 作成者に確認する必要がある。
- **Status**: Resolved — D115: hidden_facts 構造化により per-character フィルタが実装可能 (docs/spec-foundation.md §5, openspec/changes/add-state-model/)

### Q2. State Manager の state diff 候補と turn-pipeline の Commit フェーズ(非スロット)との受け渡し方式が未確定
- **対象**: `add-agent-runtime` proposal.md「What Changes」・design.md Context 冒頭。参照先: `add-turn-pipeline` specs/turn-pipeline/spec.md「スロット Protocol による差し替え可能性」要件(Simulate/Act/Resolve/**Check** の4つのみがレジストリスロット)・「Commit フェーズと commit-mode フラグ」要件(84-88行目)、tasks.md セクション6(「Commit フェーズ: Resolve/Check の結果から state diff を生成し...」というタスクが、Protocol/レジストリ経由ではなく turn-pipeline 自身の固定実装として記述されている)。
- **なぜ重要**: 本 change(修正前)は "Simulate/Act/Resolve/Commit スロットを置き換える" と記述していたが、turn-pipeline が実際に Protocol+レジストリで差し替え可能と定義しているスロットは Simulate/Act/Resolve/**Check** の4つのみであり、Commit は turn-pipeline 自身が持つ非スロットの固定ロジックである(tasks.md 6.3 で「Commit フェーズ」自身が diff を生成すると記述されている)。State Manager は resolved events から state diff 候補を生成する専用コンポーネントとして本 change に新設されているが、turn-pipeline 側に Commit をスロット化する仕組みがないため、State Manager の出力をどう Commit フェーズへ渡すか(a. Resolve スロットの出力スキーマに state diff 候補を含める、b. turn-pipeline の Commit 実装コードが State Manager を直接 import して呼び出す形に turn-pipeline 側を変更する、等)が両 change のどちらの権限にも明確に属さない状態になっている。これを決めないまま実装すると、turn-pipeline 側の Commit ロジックと agent-runtime 側の State Manager が二重に diff 生成ロジックを持つか、あるいは両者が噛み合わず接続点が存在しないまま終わる。
- **自己調査**: `add-turn-pipeline` の spec.md/design.md/tasks.md を通読し、Commit がスロットとして扱われていないことを確認した(D2/D4 の記述、tasks.md セクション3「スロット Protocol」がSimulate/Act/Resolve/Checkのみを対象にしている一方、セクション6「ターンステータスと Commit フェーズ」は非スロットの直接実装タスクとして分離されている)。`add-turn-pipeline` は本 change の依存元(先行 change)であり、そのスロット機構を本 change から変更する権限はない。
- **検討した選択肢**: A) Resolve スロットの出力型(turn-pipeline 側で Pydantic モデルとして未確定)に state diff 候補フィールドを追加し、Conflict Resolver の実装が内部で State Manager も呼び出して両方をまとめて返す形にする / B) turn-pipeline の Commit フェーズ実装(非スロット)自体を、本 change が提供する State Manager の関数を直接呼び出すよう書き換える(turn-pipeline 側コードへの変更が必要 — 本 change 単独では実施不可) / C) Commit も Simulate/Act/Resolve/Check と同様に Protocol+レジストリ化するよう `add-turn-pipeline` に修正を依頼し、State Manager をその Commit スロット実装として登録する。
- **推奨案**: C が最も turn-pipeline の既存設計思想(D108: レジストリベースの差し替え可能性)と整合的だが、`add-turn-pipeline` 自体の spec 変更を要するため本 change 単独では決定できない。次点として A(Resolve スロット出力に diff 候補を含める)は turn-pipeline 側のスロット出力スキーマがまだ実装未確定であるため、両 change の実装順序次第では成立し得る現実的な代替案。
- **不足インプット**: `add-turn-pipeline` の実装担当(または責任者)に、Commit フェーズを将来的にスロット化する意図があったか、あるいは Resolve スロット出力に state diff 候補を含める設計を許容するかの確認が必要。
- **Status**: Resolved — D113: BuildDiff スロット新設、State Manager がその実装 (docs/spec-foundation.md §6, openspec/changes/add-turn-pipeline/)

### Q3. Conflict Resolver における「衝突検出」と「roll が必要な contested」を常に同一視してよいか
- **対象**: `add-agent-runtime` specs/agent-runtime/spec.md「Conflict Resolver の衝突検出と判定要求」要件(70-79行目)。
- **なぜ重要**: 要件文は「同一 target_id を持つ複数候補または結果が排他的な候補の組」を衝突として検出し、「結果が不確実または争われている(contested)候補」について roll を要求すると書かれている。この2つの文言(「衝突」と「contested」)の関係が、常に同義(検出された衝突は全て roll を要する)なのか、それとも一部の衝突は D4 の順序ポリシー(directive優先→active_characters順→背景イベント)だけで roll なしに決定的に解決できる(例: 片方の候補が現在状態と矛盾し機械的に無効と判定できる場合)のかが未確定。この判断基準次第で、Conflict Resolver が「roll に頼らず決定的に解決できる衝突」を持つかどうかという実装方針が変わり、また物語のテンポ感(全ての対立が乱数判定を経るか、一部は自明に解決されるか)というゲームデザイン上の意図にも関わる。
- **自己調査**: design.md D4・spec.md 該当 Scenario を確認したが、「衝突」と「contested」を区別する具体例や、roll なしで解決される衝突のシナリオは spec 内に一つも存在しない(全 Scenario は検出 or roll要求のみを扱う)。企画書 §7.3(GM裁定)を参照したが、システム側の自動解決 vs ユーザー介入の裁定という別軸の話であり、この論点には直接答えていない。
- **検討した選択肢**: A) 「検出された衝突は例外なく contested として扱い、必ず roll を要求する」と単純化する(実装・テストが最も容易、YAGNI に沿う) / B) 「一方の候補が現在状態(例: 対象が既に不在・死亡)と矛盾する場合は roll なしで機械的に除外し、残った候補同士のみ contested として roll する」という2段階の解決ルールを追加する(より自然だが実装・仕様の複雑度が増す)。
- **推奨案**: A(単純化)を暫定推奨する。既存 Scenario と矛盾せず、mock provider による決定的リグレッションテスト(design.md Goals)に最も適合する。ただし B のような「明らかに不可能な候補の機械的除外」は将来の narrative quality 向上の観点で有用な可能性があり、別 change での拡張候補として残す。
- **不足インプット**: プロダクトオーナー(ユーザー)に、全ての衝突が roll を経ることを望むか、それとも一部の衝突(自明に無効な候補を含むもの)は roll なしで解決してよいかの確認が必要。
- **Status**: Resolved — D121: 全衝突を例外なく roll で解決(バッチ1) (openspec/changes/add-agent-runtime/)
