## add-state-model — Grill残課題 (20260703)

### Q1. `target: relationship` の StateDiffChange/Intervention はどの id で対象を特定するか
- **対象**: `openspec/changes/add-state-model/specs/state-model/spec.md`「Relationship モデル」要件(`RelationshipState` は `from`/`to` の有向ペアのみを持ち、独自の `id` フィールドを持たない)。および spec-foundation §5.1 の StateDiff 形式コメント(`id: char_002 # target が world/reader_state 等の単一ファイルなら省略`)。
- **なぜ重要**: `add-intervention/specs/intervention/spec.md` の Requirement「Intervention スキーマ」は `target`(StateDiff target enum に `relationship` を含む)について「対象を一意特定する任意の id を持たなければならない(SHALL)」と明記しており、`target: relationship` にも id 参照が存在する前提で書かれている。一方 `add-state-model` の `RelationshipState` には id フィールドが無く、`from`/`to` の複合キーでしか一意に特定できない。このまま実装すると、add-intervention 側が要求する「relationship の id」を add-state-model が提供できず、後続 change(add-intervention 実装時)でスキーマの手戻りが発生する。
- **自己調査**: `openspec/changes/add-state-model/specs/state-model/spec.md` の ID フォーマット検証要件(全 ID フィールドの列挙)に `relationship` が含まれていないことを確認。`docs/spec-foundation.md` §3 の ID 規約リスト(`char_001` `faction_001` `scene_001` `world_001` `event_0001` `int_0001` `roll_0001` `diff_0001` `thread_001`)にも relationship 用の型は存在しない。project_plan.md §14.5 の Relationship 例にも id フィールドは無い。spec-foundation.md は本 change から編集不可(cross-cutting)であり、正準 ID 一覧を拡張する権限が add-state-model 単独には無いため自己解決を断念した。
- **検討した選択肢**: A) `RelationshipState` に新規の sequential id(例 `rel_001`、spec-foundation §3 には無い新しい型)を追加する — spec-foundation の正準 ID 一覧を実質的に拡張することになり、本 change 単独での決定には重すぎる。 B) `from`/`to` の複合キー(例 `char_001:char_002`)を「id」として StateDiffChange/Intervention の `id` フィールドに許容する — `<type>_<zero-padded番号>` 規約と形が異なり ID フォーマット検証要件と矛盾する。 C) `target: relationship` の StateDiffChange/Intervention だけ `id` を必須にせず、代わりに `from`/`to` を直接持たせる(target 別にフィールド構成を変える) — StateDiffChange のスキーマが target ごとに分岐し、実装・検証が複雑化する。
- **推奨案**: B のバリアント(`from__to` のような明示的な複合キー文字列を正式なフォーマットとして spec-foundation に提案し、ID フォーマット検証の例外として明文化する)が実装コストと可読性のバランスが良いと考えるが、spec-foundation.md の正準 ID 一覧を変更する権限は本 change に無いため、ユーザー判断が必要。
- **不足インプット**: spec-foundation.md §3 の正準 ID 一覧に relationship 用の型(またはその代替特定方式)を追加するか、複合キー方式を正式に許容するかの意思決定。
- **Status**: Resolved — D116: 複合キー <from>__<to> を正式採用 (docs/spec-foundation.md §3, openspec/changes/add-state-model/)

### Q2. `SceneState.hidden_facts` の要素型(文字列 vs visibility 付き構造化オブジェクト)
- **対象**: `openspec/changes/add-state-model/specs/state-model/spec.md`「Scene モデル」要件(`hidden_facts` を単純な文字列リストとして扱う前提)。
- **なぜ重要**: `docs/spec-foundation.md` §4.1 は Scene スコープを「シーン参加者(hidden はスコープ指定に従う)」と定義しており、hidden_facts がキャラクターごとに異なる可視性(スコープ指定)を持ちうることを示唆している。実際 `add-agent-runtime/specs/agent-runtime/spec.md` の Requirement「Character Agent コンテキストのスコープ限定」は「本人が参加する scene の可視情報(`reader_visible_facts` および **visibility 上本人が見てよい** `hidden_facts`)」とキャラクター単位でフィルタする前提の文言になっている。project_plan.md §14.6 の例では hidden_facts は単純な文字列リストであり、per-fact の visibility/known_by を表現できない。単純な文字列リストのままでは add-agent-runtime が要求する「本人が見てよい hidden_facts」というキャラクター単位のフィルタリングを実装できず、情報スコープ(spec-foundation の最重要契約)の実装が破綻する。
- **自己調査**: `openspec/changes/add-state-model/specs/state-model/spec.md` の Scene 要件、`docs/project_plan.md` §14.6 の Scene 例、`docs/spec-foundation.md` §4.1 の Scene スコープ定義、`add-agent-runtime/specs/agent-runtime/spec.md` および `add-agent-runtime/specs/consistency-checks/spec.md`(Leak Checker は hidden_facts を「テキスト」として部分文字列一致検査するのみで、per-fact visibility の存在を前提にしていない)を確認した。両 consumer 仕様の間で hidden_facts の構造に対する前提が食い違っており(agent-runtime は構造化を要求、consistency-checks は非構造化で足りる書き方)、add-state-model が今どちらの形で確定しても一方と矛盾しうるため、自己解決を断念した。
- **検討した選択肢**: A) `hidden_facts` を `Event` と同様の `visibility`/`known_by` を持つ構造化オブジェクトのリストに変更する — spec-foundation §4.1 と add-agent-runtime の文言に整合するが、project_plan §14.6 の例・dot-path の id 一致解決規則(hidden_facts 要素に id が必要になる)・Leak Checker の実装に波及する。 B) 現状どおり `list[str]` のまま維持し、「本人が見てよい hidden_facts」は将来 add-agent-runtime 側が全 hidden_facts を一律シーン参加者に見せる(スコープ = シーン全体)という簡易解釈で吸収する — spec-foundation §4.1 の「hidden はスコープ指定に従う」という記述と整合しない可能性がある。
- **推奨案**: A(構造化オブジェクト化)が spec-foundation §4.1 と add-agent-runtime の要求に最も忠実だが、影響範囲が大きい(dot-path・Leak Checker・project_plan の例示との整合)ため、add-agent-runtime 側の実装者と合意の上で決定すべき。
- **不足インプット**: `hidden_facts`(および同様の疑問がありうる `reader_visible_facts`)をキャラクター単位でスコープ分離する必要が実際にあるか、あるとすればどの粒度(fact 単位 visibility か、シーン単位で一律か)で表現するかの決定。
- **Status**: Resolved — D115: hidden_facts を構造化 (fact_NNN/text/visibility/known_by) (docs/spec-foundation.md §5, openspec/changes/add-state-model/)

### Q3. dot-path の id 一致 `remove` が対象不在だった場合の挙動、および id 一致 `remove` の inverse diff 生成に必要なフル値の取得元
- **対象**: `openspec/changes/add-state-model/design.md`「dot-path のリスト解決」決定、および `openspec/changes/add-state-model/specs/state-model/spec.md`「inverse diff の生成と保存」要件(add change の逆操作の Scenario のみが存在し、remove change の逆操作の Scenario が無い)。
- **なぜ重要**: (1) `path が対象モデルのフィールド階層として解決可能か」という適用前検証は spec.md に明記されているが、これは path の「形」の検証であり、`remove` が指定した特定の値/id が現在のリストに実在するかどうかの検証ではない。id 一致リストに存在しない id を `remove` した場合、適用前検証エラー(reject)にするのか、apply 時エラー(他の change とまとめてアトミックに巻き戻す「対象不在」の一種)にするのか、あるいは no-op として黙って成功させるのかが明記されていない。(2) id 一致リストの要素(例: CanonEntry)を `remove, id: canon_005` で削除する場合、その StateDiffChange 自体には削除対象のフル内容(`value`)が含まれない。この `remove` から `InverseStateDiff` の `add`(削除された要素を復元するにはフル内容が要る)を生成するには、diff データ自体からではなく適用前状態(pre-state)から値を読み取る必要があるが、design.md の逆操作規則(add↔remove、set は旧値保持、delta は符号反転)はこの「remove からの inverse 生成には pre-state 参照が必要」という実装上の要件を明示していない。
- **自己調査**: `design.md`「dot-path のリスト解決」「適用前状態の保存」決定、`spec.md`「StateDiff モデルと dot-path 検証」「StateDiff の適用」「inverse diff の生成と保存」要件を確認したが、いずれも「remove 対象が既に存在しない場合」のシナリオと「id 一致 remove の逆操作としての add がフル value をどこから得るか」のシナリオが無い。これは実装方針(バリデーション設計・関数シグネチャ)を決める判断であり、既存ドキュメントの記述からは一意に導出できないため、他の細かな型未定義箇所とは異なり残課題として明示することにした。
- **検討した選択肢**: A) 適用前検証で「path 解決可能性」に加え「対象値/id の存在確認」も行い、存在しなければ検証エラー(reject)とする — 他の change と同様アトミックに拒否できるが、apply 前に current state 全体を都度スキャンする追加コストが発生する。 B) 適用時に「対象不在」を型不一致等と同列の apply-time エラー(既存の「StateDiff の適用」要件がカバーする reject 経路)として扱う — 追加のバリデーションフェーズを増やさずに済む。 C) 存在しない要素の remove を no-op として黙って許容する — D107(state 変更は全て diff 経由・監査可能性)の趣旨に反し、LLM が意図と異なる diff を出しても検出できなくなるため却下寄り。
- **推奨案**: B(適用時エラーとして既存の reject 経路に統合)が新規のバリデーションステップを増やさずシンプルであり、あわせて「id 一致 remove の StateDiffChange は `value` を省略可能だが、apply エンジンは適用前状態から削除対象のフル内容を読み取って inverse diff の `add.value` に格納する」という実装契約を design.md に明記する形を推す。
- **不足インプット**: (1) 存在しない要素への remove を reject するか no-op にするかの最終判断、(2) id 一致 remove の StateDiffChange に `value` を必須にするか省略可能にするか(省略可能なら apply エンジンが pre-state から読み取ることを正式な契約にする)の確認。
- **Status**: Resolved — apply時エラー+pre-stateからinverse値取得を正式契約化 (openspec/changes/add-state-model/)
