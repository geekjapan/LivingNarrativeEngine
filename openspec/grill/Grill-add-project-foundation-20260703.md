## add-project-foundation — Grill残課題 (20260703)

### Q1. 必須 state ファイル欠落時の扱いが add-state-model と正反対
- **対象**: `openspec/changes/add-project-foundation/specs/project-workspace/spec.md` の Requirement「必須 state ファイルの存在確認」(Scenario: `gm_vault.yaml` が存在しない workspace の読み込み → 読み込み失敗・欠落一覧を報告) と、`openspec/changes/add-state-model/specs/state-model/spec.md` の Requirement「WorldStateBundle による workspace 全体ロード」(Scenario: 同じく `gm_vault.yaml` が存在しない workspace を `StateStore.load()` に渡す → ロード成功・`WorldStateBundle.gm_vault` は空リスト)
- **なぜ重要**: 同一シナリオ(`gm_vault.yaml` 欠落)に対し、add-project-foundation は「ロード失敗」、add-state-model は「ロード成功・空扱い」と正反対の契約を定めている。`add-turn-pipeline` の Load フェーズがどちらの API を呼ぶか(あるいは両方を直列に呼ぶか)が仕様上未確定なため、実装者ごとに「workspace 検証は fail-fast にすべきか lenient にすべきか」の解釈が割れる。特に `add-cli-and-sample` の `status`/`turn` コマンドがエラー時にどう振る舞うべきかに直結する。
- **自己調査**: 両 change の spec.md・proposal.md・design.md を確認。add-project-foundation 側は「ファイル内容そのもののスキーマ検証(add-state-model の範囲)は行わない」と明記し、両者を役割分担された別関数として位置づけているが、「ファイルの存在」という同一の判定基準について矛盾する契約を持つ点は解消されていない。add-state-model 側の spec.md はこの矛盾に一切言及していない(add-project-foundation への参照なし)。両 spec とも既に確定した change proposal であり、add-project-foundation 側だけを書き換えても矛盾は解消しない(state-model 側が変わらない限り「同じ7ファイルの欠落」に対する挙動の一方が誤りとして残る)。
- **検討した選択肢**: A) add-project-foundation の「プロジェクト読み込み API」を size-model の StateStore とは独立した「診断用・非ゲーティングのレポート機能」として明示的に格下げし、パイプラインの Load フェーズは常に add-state-model 側(lenient)を正とする。B) 逆に add-state-model 側を「必須7ファイルは fail-fast、それ以外(scenes/characters 配下等)のみ lenient」に狭め、必須ファイルについては add-project-foundation の契約を正とする。C) 両者を統合し、`add-project-foundation` の存在確認 API 自体を廃止して `add-state-model` の StateStore.load 一本化に委譲する(必須/任意の区別自体をなくす)。
- **推奨案**: B が妥当と考える。理由: spec-foundation §5 は該当7ファイルを workspace レイアウトの正規の一部として列挙しており(Appendix C も同様)、`init` は必ずこれらを生成するため、通常運用下で「意図せず必須ファイルが消えている」状態は「壊れたワークスペース」であり fail-fast で気づけた方が安全(spec-foundation §6 の失敗ポリシー「決して黙って握り潰さない」の精神とも整合)。一方 `characters/*.yaml` や `scenes/*.yaml` のような可変個数ファイル群については元々 add-project-foundation 側も「必須ファイル」として個別チェックしておらず、add-state-model の lenient 扱いは妥当。
- **不足インプット**: `add-turn-pipeline` の Load フェーズが実際にどちらの API を呼ぶ設計か(両方叩くのか、片方のみか)をユーザーに確認する必要がある。また、この矛盾を解消するには `add-state-model` の spec.md(既に固定された change)を編集する権限が必要であり、本 grill の対象範囲外(add-project-foundation のみ編集可)のため、user 確認の上で `add-state-model` 側の change も別途修正が必要。
- **Status**: Resolved — D117: 必須7ファイルは fail-fast で統一、state-model 側を修正 (openspec/changes/add-state-model/)

### Q2. `init` コマンドの所有権が project-workspace と cli (add-cli-and-sample) の間で未整理
- **対象**: `openspec/changes/add-project-foundation/specs/project-workspace/spec.md` の Requirement「init コマンドによるプロジェクト作成」(`--title` のみ必須、固定既定値で残りを埋める版) と、`openspec/changes/add-cli-and-sample/specs/cli/spec.md` の Requirement「`init` によるプロジェクト新規作成とテンプレート検証」(`--title --genre --tone --template --output` を受け取り `mist_station`/`minimal` テンプレートを提供する、より完成度の高い版)
- **なぜ重要**: 同じ CLI コマンド `living-narrative init` の contract が2つの異なる capability(`project-workspace` と `cli`)にそれぞれ独立した Requirement として存在する。`add-cli-and-sample` の proposal.md は Capabilities 節で `project-workspace` を「Modified Capabilities」として宣言しておらず、実装者からは「add-project-foundation の init 実装を add-cli-and-sample が完全に上書き・置換する」のか「project-workspace 側の init も別途生き残り、cli 側が別の何かを追加する」のか読み取れない。DAG 上どちらの spec が最終的な正本かが不明確なまま両方が「SHALL」で確定要件として存在しているため、実装順序によっては二重実装や、後から来た変更が前の契約を静かに破る事故が起きうる。
- **自己調査**: add-project-foundation の design.md に「`add-cli-and-sample` が同じ `init` サブコマンドに `--genre`/`--tone`/`--template`/`--output` を追加し、テンプレート選択を導入する計画であり、本 change の `init` はその前段の最小実装として位置づける」という前方互換の注記を追加し、少なくとも add-project-foundation 側の読者には「後で置き換わる暫定実装」であることを明示した(本 grill で自己解決済み)。しかし、これは add-project-foundation 側のドキュメントのみの修正であり、`add-cli-and-sample` の proposal.md/spec.md 側に「project-workspace の init Requirement を置き換える」ことを示す正式な宣言(Modified Capabilities への `project-workspace` 追加、または `init` Requirement 同士の明示的な関係付け)がない限り、OpenSpec のカバレッジ上は矛盾が残る。`add-cli-and-sample` は編集対象外(cross-cutting)のため、ここでは自己解決できない。
- **検討した選択肢**: A) `add-cli-and-sample` の proposal.md の Capabilities 節に「Modified Capabilities: `project-workspace`(`init` コマンドの CLI フラグ・テンプレート選択を拡張)」を追加し、spec delta として project-workspace 側の `init` Requirement を「MODIFIED」として明示する。B) 現状のまま(`cli` capability 側の `init` Requirement が事実上の最終正本であり、`project-workspace` 側は「`init` という名のコマンドが存在すること」の暫定契約に過ぎないとみなす)運用ルールを spec-foundation か DAG ドキュメントに明記する。C) `project-workspace` 側の `init` Requirement 自体を「`add-cli-and-sample` 完了後は `cli` capability の該当 Requirement に置き換えられる暫定実装」と明示的に注記し、正本の所在を spec 内テキストで宣言する(design.md には追記済みだが spec.md 本体には未反映)。
- **推奨案**: A が最も OpenSpec の設計と整合する(capability の変更履歴が spec delta として追跡可能になる)。ただし本 grill の実行者は `add-project-foundation` のみ編集可能なため、この修正は `add-cli-and-sample` 側で別途行う必要がある。
- **不足インプット**: `add-cli-and-sample` の proposal.md を編集する権限とユーザーの承認、および「project-workspace の init Requirement を廃止/置換するか、両立させるか」の意思決定。
- **Status**: Resolved — add-cli-and-sample が project-workspace を MODIFIED 宣言 (openspec/changes/add-cli-and-sample/)
