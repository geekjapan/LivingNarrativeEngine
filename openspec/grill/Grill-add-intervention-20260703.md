## add-intervention — Grill残課題 (20260703)

### Q1. Permission table のownershipが `add-intervention` と `add-session-autonomy` で二重化している
- **対象**: `openspec/changes/add-intervention/specs/intervention/spec.md` Requirement「Role Permission Hook」および `design.md` D2 / `openspec/changes/add-session-autonomy/specs/session-autonomy/spec.md` Requirement「ユーザーモードの権限マトリクス」
- **なぜ重要**: 両 spec とも「許可されない user_mode の介入は拒否する」ことを SHALL で規定しているが、実装のタイミング・データソースが異なる。`add-intervention` は Intervention *生成時*(Interpreter/直接入力の両経路)に `check_permission(type, user_mode)` で拒否すると明記しており、生成に成功した Intervention は既に許可済みという前提になる。一方 `add-session-autonomy` は「Intervention Interpreter から渡された場合、Session Autonomy はその介入を拒否する」と記述しており、Intervention が一度生成された後に別の場所で再度拒否されることを前提にしている。さらに、`add-intervention` の spec.md は15型中 `canon_edit`/`hidden_truth_edit` の2行のみを SHALL レベルで具体的に規定しており、残り13行の許可 user_mode 集合は仕様上未確定(design.md D2 は「15種全ての行を明示的に埋める」と言うのみで値を規定しない)。対して `add-session-autonomy` は15型×6モードの完全なマトリクスを独自に SHALL として持つ。加えて spec-foundation §1.2 の DAG では `add-intervention` → `add-session-autonomy` の順であり、`add-intervention` は `add-session-autonomy` に依存できない(proposal.md の Dependencies にも session-autonomy は無い)。そのため `add-intervention` 単独で実装する際、残り13行の許可集合をどう埋めるかの正本データが無く、後から実装される `add-session-autonomy` のマトリクスと食い違う実装が生まれるリスクがある。
- **自己調査**: `add-intervention` の proposal.md/design.md/spec.md/tasks.md 全文、`add-session-autonomy` の spec.md 該当 Requirement、spec-foundation §1.2 の DAG 順序を確認した。`add-intervention` の Dependencies セクションに `add-session-autonomy` が含まれないこと、Modified Capabilities が「なし」であることも確認した。二重化そのものは cross-change の設計判断であり、`add-intervention` の change フォルダ内の編集だけでは解消できない(相手側の spec.md を編集する権限が無い、かつ `add-session-autonomy` はまだ archive されていないため MODIFIED Requirements で正式に統合することもできない)。
- **検討した選択肢**: A) `add-intervention` の permission table を唯一の正本とし、`add-session-autonomy` 側の「ユーザーモードの権限マトリクス」要件は intervention capability の `check_permission` を呼び出すだけの薄いラッパーとして書き換える(値は intervention 側の15行に統一)。 B) 逆に `add-session-autonomy` を正本とし、`add-intervention` の「Role Permission Hook」要件を「type ごとの許可集合は `session-autonomy` から供給される」という抽象インターフェースのみに弱め、`add-intervention` 単体では2行の最低制約(canon_edit/hidden_truth_edit → full_gm/god)のみを保証する。 C) 両方を維持しつつ、`add-intervention` の実装時点(session-autonomy 未実装)では全13行を最も緩い制約(例: god のみ拒否なし)にしておき、`add-session-autonomy` 実装時に必ず両テーブルの値を突き合わせる検証タスクを追加する。
- **推奨案**: B。理由: spec-foundation §1.2 の DAG 順序に従えば、実際に user_mode ごとの許可を「正しく」判断できる情報(9.1-9.5 の autonomy level、8.1-8.6 のモード別できること一覧)は企画書 §8 起点であり、`add-session-autonomy` が正本になる方が自然(add-session-autonomy 側の spec.md 自身も「介入タイプの列挙とスキーマ自体は intervention capability が正本」と明記しており、値の正本は元々 session-autonomy 側にあると読める)。`add-intervention` は D107 のような普遍的制約(canon_edit/hidden_truth_edit は god 系のみ)だけを自前で持ち、それ以外は「許可判定はプラガブルであり、既定では全許可」という設計にすれば、DAG の後段で session-autonomy が本来の制限を追加する形に自然に収まる。
- **不足インプット**: どちらの capability が permission table の正本になるべきかは、単一 capability の grill では決定不可能な設計判断。ユーザー(またはプロジェクトオーナー)に、A/B/C いずれの方針を取るか確認する必要がある。
- **Status**: Resolved — D114: session-autonomy を正本、intervention は普遍不変条件+プラガブル判定 (openspec/changes/add-intervention/, openspec/changes/add-session-autonomy/)

### Q2. `reveal_control` の Commit フェーズ昇格制御を呼び出す統合点が `add-turn-pipeline` 側に存在しない
- **対象**: `openspec/changes/add-intervention/specs/intervention/spec.md` Requirement「reveal_control の意味論」/ `design.md` D4 / `openspec/changes/add-turn-pipeline/specs/turn-pipeline/spec.md` Requirement「Commit フェーズと commit-mode フラグ」
- **なぜ重要**: `add-intervention` は「`must-not-reveal` としてマークされた事実は Commit フェーズにおいて Reader State への昇格をブロックされなければならない(SHALL)」と規定しているが、Commit フェーズ自体を定義・実装するのは `add-turn-pipeline` であり、その spec は「Commit フェーズは Resolve/Check の結果から state diff を生成し、state-model の diff 適用インターフェースへ渡さなければならない」としか言っておらず、intervention capability のデータを参照する義務は一切無い。`add-intervention` の proposal.md の Modified Capabilities は「なし」のままであり、`add-turn-pipeline` の Commit フェーズ要件を変更する提案にもなっていない。このままでは、`add-turn-pipeline` を素直に実装した場合、reveal_control の SHALL 要件を満たすコードパスがどこにも存在しないことになる。
- **自己調査**: `add-intervention` の proposal.md(Modified Capabilities セクション)、design.md D4、spec.md「reveal_control の意味論」要件、`add-turn-pipeline` の spec.md「Commit フェーズと commit-mode フラグ」要件、`add-agent-runtime` の「State Manager の state diff 候補生成と検証」要件を確認した。加えて `openspec list --specs` で確認した通り、本リポジトリではどの capability もまだ archive されておらず main spec が存在しないため、`add-intervention` から `add-turn-pipeline` へ `## MODIFIED Requirements` で正式にフックを追加する OpenSpec 上の手段が無い(archive 前の並行 proposal 同士では spec delta の適用先が存在しない)。そのため本課題は `add-intervention` の change フォルダ内の編集だけでは解消できない。
- **検討した選択肢**: A) `add-intervention` の proposal.md の Modified Capabilities に `turn-pipeline` を追加し、`add-turn-pipeline` が archive された後に本 change の実装時点で turn-pipeline 側の Commit フェーズ要件へ「reveal_control フィルタの呼び出し」を追記する前提で進める(実装順序への依存を明記するのみ)。 B) 統合点を turn-pipeline の Commit フェーズではなく `add-agent-runtime` の State Manager(state diff 候補生成)側に移し、State Manager が候補生成時に intervention capability の reveal_control フィルタを呼び出す設計に変更する(State Manager は既に「全状態を参照できる」ため技術的には可能)。 C) turn-pipeline 側に汎用的な「Commit 前フィルタ」拡張点(Protocol + レジストリ、D108 と同じ思想)を用意してもらい、intervention capability がそこに reveal_control フィルタを登録する。
- **推奨案**: C。理由: D108 の「拡張点は Protocol + レジストリ」という既存方針と一貫しており、reveal_control 以外の将来の Commit 前フィルタ(例: 将来バッチでの stop_condition 系介入)にも再利用できる。ただし turn-pipeline 側の spec 変更が必要なため、`add-turn-pipeline` の change 提案者との調整が要る。
- **不足インプット**: `add-turn-pipeline` が現時点で archive 済みかどうか、archive 済みであれば `add-intervention` から MODIFIED Requirements で正式にフックを追加できるため、まずその状態を確認してほしい。未 archive であれば、どちらの change がこの統合点を先に定義するかの順序をユーザーに決めてもらう必要がある。
- **Status**: Resolved — D113: BuildDiff スロット契約で強制、turn-pipeline への hook 不要 (docs/spec-foundation.md §6)

### Q3. `interventions.yaml` の履歴エントリが `rerun_turn` 後も旧ターンの event/diff id を指したまま残る可能性
- **対象**: `openspec/changes/add-intervention/specs/intervention/spec.md` Requirement「Intervention 履歴インデックス」/ `openspec/changes/add-session-autonomy/specs/session-autonomy/spec.md` Requirement「Rerun の乱数消費セマンティクス」
- **なぜ重要**: 本 grill での自己解決により、`interventions.yaml` の各エントリは当該ターンの Commit フェーズ完了時点で event id / state diff id を確定させて一度だけ書き込む、という書き込みタイミングの曖昧さは解消した。しかし `add-session-autonomy` の `rerun_turn` は「破棄された旧ターンの artifact は削除せず、監査可能な形で保持しなければならない(SHALL)」としており、同一ターン番号に対して新しい event/diff id 列で再実行が行われ得る。この場合、既に Commit 完了時点で書き込み済みの `interventions.yaml` エントリが、破棄された(もはや state に反映されていない)旧ターン実行の event/diff id を指したまま残るのか、rerun 後に新しい参照へ更新されるのか、あるいは旧エントリと新エントリが両方併存するのかが、両 capability のどちらの spec にも規定されていない。
- **自己調査**: `add-intervention` の spec.md「Intervention 履歴インデックス」要件(本 grill で追記した部分含む)、`add-session-autonomy` の「Rerun の乱数消費セマンティクス」要件、「GM レビューゲートのフロー」要件を確認した。`rerun_turn` は session-autonomy が所有する概念であり、その挙動(artifact の保持方法、同一ターン番号での再実行が新規ディレクトリになるのか上書きになるのか)自体が session-autonomy 側でも完全には規定されていないため、`add-intervention` 単独でこの相互作用を確定させることはできない。
- **検討した選択肢**: A) `interventions.yaml` のエントリに `superseded_by_rerun: bool` 相当のフラグを持たせ、rerun 発生時に旧エントリを論理的に無効化しつつ新エントリを追記する(既存エントリの内容は変更しない=「上書きしない」制約を維持)。 B) `interventions.yaml` は元の intervention と元の(最初の)実行結果のみを記録する不変ログとし、rerun 後の真の適用状態は `review.yaml`(session-autonomy が所有)側でのみ追跡する — つまり `interventions.yaml` の完全性より単純さを優先する。 C) rerun 時に同一ターン番号で `interventions.yaml` への追記自体を差し戻し不可能にする(rerun 前に Commit が完了していなければ、そもそも `interventions.yaml` へはまだ書き込まれていないはずという運用ルールにする)。
- **推奨案**: A。理由: 本 change の核心原則(企画書 §5.2、ユーザー指示を消してはならない)を rerun 後も維持できる。ただし `superseded_by_rerun` フィールドの具体的な意味論と、それをどのタイミングで誰が設定するか(intervention capability か session-autonomy capability か)は cross-change の設計判断が必要。
- **不足インプット**: `rerun_turn` 実行時に同一 `turn_NNNN` ディレクトリが上書きされるのか、別ディレクトリ(例 `turn_0018_rerun_1`)に保持されるのかという `add-session-autonomy` 側の未規定事項を先に確定させる必要がある。これはユーザーまたは `add-session-autonomy` の change 提案者への確認事項。
- **Status**: Resolved — D112: discarded 退避 + superseded_by_rerun フラグ (openspec/changes/add-intervention/)
