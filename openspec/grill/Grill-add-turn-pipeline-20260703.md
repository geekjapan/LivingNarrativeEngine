## add-turn-pipeline — Grill残課題 (20260703)

### Q1. Act フェーズのスキーマ検証リトライ失敗後、ターンステータスは `failed` か `stopped_for_review` か
- **対象**: `specs/turn-pipeline/spec.md` の Requirement「スキーマ不一致時のretry委譲」
- **なぜ重要**: spec-foundation §6「失敗ポリシー: schema 不一致 → 最大2回 retry → `stop_for_review`」および §8「検証失敗 → 修正指示付き retry(最大2回)→ 失敗時 stop_for_review」は、schema 不一致の retry 枯渇を明示的に `stop_for_review`(=`stopped_for_review`)としているが、turn-pipeline の現行 spec.md は同じ状況を `failed` としている。どちらを実装するかで、CLI の表示メッセージ・GM への提示要否・add-session-autonomy の停止条件マッピング(`checker_error` 等 9 条件のいずれにも Act スキーマ失敗は含まれていない)が変わる。
- **自己調査**: spec-foundation §6/§8(両方とも同一表現)、turn-pipeline の該当 Requirement・Scenario、llm-provider spec の「retry上限到達時の型付きエラー」(「具体的なターンステータス名へのマッピング規則は本 capability の責務外」と明記され、判断はturn-pipeline側に委ねられている)を確認した。自己解決できなかった理由: `stopped_for_review` は他の全 Requirement(Check フェーズのエラー検出、Commit フェーズ)で「state diff が生成されているが未適用」という状態と結び付けて定義されており、Act フェーズ失敗時点では Resolve/Narrate/Check/Commit が未実行で state diff が存在しない。spec-foundation の一文だけを根拠に turn-pipeline 側の定義を書き換えると、`stopped_for_review` の意味論(diff の存在を前提とするか否か)に新たな内部矛盾を生む可能性があり、確信を持って自己解決できなかった。
- **検討した選択肢**: A) spec-foundation の文言どおり `stopped_for_review` に統一する(diff 不在でも stopped_for_review を許容する形に「ターンステータスモデル」の定義を緩める) / B) 現行どおり `failed` を維持し、spec-foundation §6/§8 の当該一文を「大まかな要約であり、正確な状態名を意図したものではない」と解釈する(spec-foundation は本 change の対象外につき編集不可)。
- **推奨案**: B寄り。turn-pipeline 内では Check フェーズ完了後の「diff は生成済みだが未適用」という一貫した意味論を `stopped_for_review` に持たせており、Act フェーズのような「diff 自体が生成されない」失敗は既存の「失敗時の部分artifact永続化」Requirement(あらゆるフェーズ例外 → `failed`)と自然に整合する。ただし spec-foundation 側の記述との食い違いは正式に解消されるべき。
- **不足インプット**: spec-foundation §6/§8 の「stop_for_review」という語が canonical enum 値 `stopped_for_review` を厳密に指しているのか、それとも「エラーを黙って握り潰さずに人間の関与を要求する」という一般的な意味での言い回しなのかのユーザー判断。後者であれば turn-pipeline の現行 `failed` 採用のままで矛盾は解消する。
- **Status**: Resolved — D110: failed を正とし spec-foundation §6/§8 を修正済み (docs/spec-foundation.md)

### Q2. `commit_mode` フィールドは project.yaml のスキーマ正本(project-workspace capability)に正式追加されるべきか
- **対象**: `specs/turn-pipeline/spec.md` の Requirement「Commitフェーズとcommit-modeフラグ」、design.md D4
- **なぜ重要**: `add-project-foundation` の project-workspace spec は project.yaml の Pydantic モデルに含めるべきフィールドを列挙しているが `commit_mode` は含まれていない。同 spec は「未知フィールドはロードを失敗させてはならず、警告として扱わなければならない」とのみ規定しており、警告対象フィールドの値が Pydantic モデル上に保持される(`extra="allow"` 相当)のか、警告のみで値自体は破棄される(`extra="ignore"` 相当)のかを規定していない。後者であれば、project.yaml に `commit_mode: auto` と書いても、project-workspace のロード API 経由で読み込んだ後にはその値を turn-pipeline から参照できない可能性がある。
- **自己調査**: spec-foundation §5(project.yaml フィールド一覧に `commit_mode` なし)、add-project-foundation の「project.yaml スキーマ」「project.yaml ロード時検証とエラー集約」Requirement、turn-pipeline design.md D4(「project.yaml (または turn 実行時オプション)」という曖昧な言い回し)を確認した。turn-pipeline の spec.md 自体は「プロジェクト設定の commit-mode フラグ」と明記しており project.yaml 前提で書かれている。自己解決できなかった理由: 正しい修正は project-workspace capability(add-project-foundation)のスキーマ Requirement を拡張することであり、これは sibling change の編集にあたるため本 change の範囲外(cross-cutting)。
- **検討した選択肢**: A) add-project-foundation 側で `commit_mode`(既定値 `auto`)を project.yaml スキーマの正式フィールドとして追加する / B) turn-pipeline は project.yaml を経由せず、CLI 実行時オプション(例: `--commit-mode`)としてのみ commit_mode を受け取る設計に変更し、project.yaml への永続化自体をやめる(design.md の「または turn 実行時オプション」案) / C) project-workspace 側の「未知フィールド」処理を `extra="allow"` として明文化し、turn-pipeline はモデルの動的属性として commit_mode を読み取る。
- **推奨案**: A。`add-session-autonomy` が最終的にこのフラグを autonomy_level/user_mode ベースの判断へ置き換える設計(D4 の移行経路)であることを考えると、`commit_mode` も `autonomy_level`/`user_mode` と同様に project.yaml の永続設定として扱うのが一貫している。
- **不足インプット**: add-project-foundation 側のスキーマ Requirement 更新をどのタイミングで行うか(turn-pipeline 実装前に先行 change として当てるか、turn-pipeline 実装時に同一 PR で project-workspace の spec delta も追加するか)のユーザー判断。
- **Status**: Resolved — D118: commit-mode はランタイムパラメータ、project.yaml に追加しない (openspec/changes/add-turn-pipeline/)

### Q3. add-agent-runtime の proposal.md が「Simulate/Act/Resolve/Commit スロット」を置き換えるとしているが、turn-pipeline が定義する差し替え可能スロットは「Simulate/Act/Resolve/Check」の4つであり Commit は含まれない
- **対象**: `openspec/changes/add-agent-runtime/proposal.md`(「これらの実装で add-turn-pipeline が定義した Simulate/Act/Resolve/Commit スロットの組み込み最小実装を置き換える」)と、turn-pipeline `specs/turn-pipeline/spec.md` の Requirement「スロット Protocol による差し替え可能性」「組み込み最小スロット実装」(Simulate/Act/Resolve/**Check** の4スロットのみを定義し、Commit はスロットとして扱っていない)
- **なぜ重要**: agent-runtime の5コンポーネント(Context Builder / Character Agent / World Simulator / Conflict Resolver / State Manager)のうち、State Manager は「resolved event 列から state diff 候補を生成する」役割であり、これは turn-pipeline の Commit フェーズ本体のロジック(「Commit フェーズは Resolve/Check の結果から state diff を生成し…」)と重複しているように見える。しかし turn-pipeline は Commit をプラグイン可能なスロットとして設計しておらず(D2: スロットは Simulate/Act/Resolve/Check のみ)、Check スロットの実装置き換えについても agent-runtime の proposal 本文には記載がない(Check の担当は同じ change 内の別 capability `consistency-checks` になっている)。このままでは、agent-runtime 実装時に「State Manager の出力を turn-pipeline のどの差し替えポイントに接続するか」(Commit フェーズ内部ロジックの置き換えを想定しているのか、Resolve/Check スロットの出力として扱うのか)が定義されないまま実装が進むリスクがある。
- **自己調査**: turn-pipeline の proposal.md/design.md/spec.md(スロットは Simulate/Act/Resolve/Check の4つと3箇所で明記)、agent-runtime の proposal.md・spec.md(Context Builder/Character Agent/World Simulator/Conflict Resolver/State Manager の5コンポーネント、Check スロットへの言及なし)を確認した。自己解決できなかった理由: agent-runtime 側の記述誤り(または未確定の設計)であり、修正は sibling change(add-agent-runtime)の編集にあたるため本 change の範囲外。
- **検討した選択肢**: A) agent-runtime の記述を「Simulate/Act/Resolve/**Check** スロット」に訂正し、State Manager の diff 候補生成は Commit フェーズ内部(スロットではなく turn-pipeline 本体ロジック)へのフックとして別途定義する / B) turn-pipeline 側で Commit フェーズも将来的にスロット化できるよう D2 のスロット集合を Simulate/Act/Resolve/Check/Commit の5つに拡張する(現時点では turn-pipeline の設計方針そのものの変更になるため大きい)。
- **推奨案**: A。turn-pipeline は「Commit は state-model の diff 適用 API を直接呼ぶ固定ロジックであり、スロットではない」という設計を明確に貫いており(D2 参照)、これを変更する理由は今のところ見当たらない。agent-runtime 側の proposal.md の表記ミスとして扱い、State Manager の役割を「Resolve スロットまたは Commit フェーズが呼び出す diff 生成ヘルパー」として再定義するのが自然。
- **不足インプット**: agent-runtime 実装者(または `/opsx:apply` 実行時)による agent-runtime proposal.md の訂正と、State Manager コンポーネントの正確な接続点(Resolve スロットの一部として実装するか、Commit フェーズが呼び出す非スロット関数として実装するか)の確定。
- **Status**: Resolved — D113: BuildDiff スロット新設、State Manager がその実装 (openspec/changes/add-turn-pipeline/, docs/spec-foundation.md)

### Q4. session-autonomy の GM レビューゲート(accept_all/reject_all/partial/edit/rerun_turn)は turn-pipeline のどの API 経由で実行されるのか
- **対象**: turn-pipeline design.md D4(「Commit フェーズが『apply するか否かの bool』を受け取る」というインターフェース契約)と、`add-session-autonomy` の Requirement「GM レビューゲートのフロー」(`partial`/`edit`/`rerun_turn` を含む5種の決定)
- **なぜ重要**: D4 が明示する turn-pipeline 側の恒久インターフェースは「Commit フェーズが bool を受け取るだけ」という、ターン実行中に1回だけ行われる単純な apply/hold 判断である。しかし session-autonomy が要求する GM レビューゲートは、`pending_review` 状態になった**既存の**ターンに対して、後から `partial`(diff の部分適用)・`edit`(diff 編集後の再検証・適用)・`rerun_turn`(ターン全体の artifact 破棄と再実行)を行う、ターン実行そのものとは別のオペレーションである。turn-pipeline はこれらの「実行後のターンに対する事後操作」用の公開 API を一切定義しておらず、session-autonomy がこれを turn-pipeline 経由で行うのか、state-model の diff 適用 API(partial apply 等、既に state-model 側に存在する)を直接呼び出して turn-pipeline を完全にバイパスするのかが不明である。特に `rerun_turn` はターン artifact 全体の破棄・再実行を伴い、turn-pipeline の「次ターン番号の決定」ロジック(本 change で明確化した「failed ターンは同じ番号で再実行される」)と類似の番号再利用セマンティクスを必要とするが、これが turn-pipeline 側に公開 API として存在するかは spec に記載がない。
- **自己調査**: turn-pipeline design.md D4 の全文、session-autonomy の proposal.md(Impact:「統合ポイント: turn pipeline の Intervene フェーズ…、Check→Commit 境界(停止条件評価点)」— レビューゲート実行そのものの統合ポイントには言及なし)、session-autonomy spec.md の「GM レビューゲートのフロー」「Rerun の乱数消費セマンティクス」を確認した。自己解決できなかった理由: これは turn-pipeline が「今は約束していない」公開 API の要否に関わる設計判断であり、session-autonomy(sibling, 未実装)側の実装方針にも依存するため、turn-pipeline 単独では確定できない。
- **検討した選択肢**: A) turn-pipeline に「pending なターンに対して partial/edit な diff を確定 apply する」ための専用 API(例: `finalize_pending_turn(turn_number, changes_or_edited_diff)`)と「rerun_turn 用にターン番号を再利用して artifact を破棄・再実行する」API を追加する / B) turn-pipeline は関与せず、session-autonomy が state-model の diff 適用 API(partial apply・InverseStateDiff によるロールバック等、既存)を直接呼び出し、turn artifact(`review.yaml` 等)の書き込みも session-autonomy 自身が担当する設計とする。
- **推奨案**: B寄り。D4 の「TurnPipeline 側のインターフェースは変えない」という移行方針(bool のみ)を素直に読むなら、レビューゲートの事後操作は turn-pipeline の外側(session-autonomy が state-model と turn artifact を直接操作する層)で完結させる設計の方が、turn-pipeline の責務を「1ターンを1回、順序どおり実行する」ことに閉じ込められて一貫する。ただし rerun_turn のターン番号再利用ロジックは turn-pipeline が本 change で定義したもの(本 grill での自己解決分)と重複するため、少なくともその部分は turn-pipeline のユーティリティとして再利用可能な形で公開しておくことが望ましい。
- **不足インプット**: session-autonomy 実装時に、レビューゲートの事後操作が turn-pipeline のどの関数/モジュールを呼び出すのか(または全く呼び出さず state-model のみに依存するのか)のユーザーまたは session-autonomy 側設計者による確定。
- **Status**: Resolved — レビューゲート事後操作はパイプライン外(state-model API + turn-pipeline 公開ユーティリティ)で実施 (openspec/changes/add-turn-pipeline/)
