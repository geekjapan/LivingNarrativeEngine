## add-session-autonomy — Grill残課題 (20260703)

### Q1. `stop_condition` 介入タイプと企画書§10.6の10件目の停止条件の対応関係が未定義
- **対象**: `openspec/changes/add-session-autonomy/specs/session-autonomy/spec.md` Requirement「停止条件の判定とレベル別適用」/ `openspec/changes/add-intervention/specs/intervention/spec.md` Requirement「Type 別ハンドリング状況の明示」
- **なぜ重要**: `docs/project_plan.md` §10.6 は自律進行の停止条件を10件列挙しているが、本 change の9条件(character_death/major_canon_change/relationship_threshold_crossing/major_secret_reveal/checker_error/leak_suspicion/heavy_roll_failure/scene_end/target_turn_count_reached)には「ユーザー確認が必要な分岐」が含まれない。一方、本 change の権限マトリクスは `assistant_gm`/`full_gm` に `stop_condition` という介入タイプを許可している(intervention 側の type 一覧に存在)が、add-intervention 側の spec は `stop_condition` を「専用ハンドラを持たない6種」の1つとして明記しており(受理・保存され agent への constraints 提示のみ)、実際にターンを停止させる効果は本 change のどこにも定義されていない。つまり「ユーザーが任意のタイミングで停止を要求する」という10件目の停止条件を実現する手段が、type としては存在するのに配線されていない。実装者は (a) `stop_condition` 介入を無効な飾り enum のまま残す、(b) 独自に「常に即時停止させる」実装を追加する、のいずれかを選ばざるを得ず、後者を選んだ場合 `watch`/`god` レベルでもユーザー起因の停止要求を尊重すべきか(§10.6 の他条件と同様ログのみにすべきか)という規定も無い。
- **自己調査**: 本 change の proposal.md/design.md/spec.md/tasks.md 全文、`docs/project_plan.md` §10.6・§10.7、`add-intervention` の spec.md「Type 別ハンドリング状況の明示」要件・「パイプライン統合」要件を確認した。`stop_condition` の実行効果を定義する Requirement が本 change に存在しないこと、design.md の Open Questions にもこの論点が無いことを確認済み。`stop_condition` の効果は Intervene フェーズで受理された介入を Check→Commit 境界の停止条件評価(design.md D2)に接続する話であり、その評価ロジックの所有者は本 capability であるため、本来は本 change 側で定義すべき事項だが、「ユーザー確認が必要な分岐」という抽象的な文言をどう機械的なトリガーに落とすかは製品判断を要し、自己解決の範囲を超える。
- **検討した選択肢**: A) `stop_condition` 介入が確定したターンは、他の9条件と同じ枠組みで10件目の条件として扱い、`assist`/`auto` はもちろん `watch`/`god` でも(ユーザーが明示的に要求した停止である以上)常に停止させる。 B) `stop_condition` は`watch`/`god` の3件固定リスト(checker_error/scene_end/target_turn_count_reached)には加えず、他の6条件と同様ログのみとする(god の「全ガードレール解除」という設計意図(design.md D3)を優先)。 C) `stop_condition` 介入には効果を持たせず、企画書§10.6の10件目は単に本バッチでは未実装として明示的に Non-Goal に落とす。
- **推奨案**: A。理由: `stop_condition` は他の8条件(システムが検知する客観条件)と異なり、ユーザーが能動的に発行する介入であり、権限マトリクス上すでに assistant_gm/full_gm に許可されている。ユーザーが明示的に「ここで止めて」と発行した介入を god/watch レベルが黙って無視する設計は、D107 の「決して黙って握り潰さない」という一般原則や、God Mode がそれでも checker_error では停止する非対称性(spec.md 既存 Scenario)と整合しない。
- **不足インプット**: `stop_condition` 介入の実行効果(常時停止か、条件付きログのみか)と、autonomy level 別の適用有無をユーザーに確認する必要がある。確定後、本 change の「停止条件の判定とレベル別適用」Requirement に10件目として追記する想定。
- **Status**: Resolved — D119: 10番目の停止条件として全レベルで停止 (openspec/changes/add-session-autonomy/)

### Q2. `reject_all` 後のターンのナレーション・イベントが正史(canon/リプレイ出力)として扱われるかが未定義
- **対象**: `openspec/changes/add-session-autonomy/specs/session-autonomy/spec.md` Requirement「GM レビューゲートのフロー」(本 grill で追記した turn_status 写像)/ `docs/project_plan.md` §10.11・§10.12・§17.1(小説原案化)
- **なぜ重要**: 本 grill の自己解決により、`reject_all` はターンステータスを「変更ゼロ件の `applied`」として確定させ、add-turn-pipeline の「次ターン番号の決定と未解決ターンによるブロック」(解決は applied/failed のみ)という技術的制約は満たせるようにした。しかし、この写像は state diff が適用されないことのみを規定しており、当該ターンで既に生成済みの `narration.md`・`events.yaml`・`rolls.yaml` が「実際に起きた出来事」としてログ・リプレイ・小説原案化(企画書§10.11/§10.12/§17.1)の対象に含まれ続けるのか、それとも reject_all は当該ターンの物語内容そのものを「無かったこと」として export-replay から除外すべきなのかが、本 change にも export-replay(add-cli-and-sample)側にも規定が無い。これは技術的な turn_status の話ではなく、「GM が却下した展開が読者向けの正史に残るか」という物語上の意味論であり、実装方針次第で `export replay` の出力内容が変わる。
- **自己調査**: 本 change の spec.md 全体、`docs/project_plan.md` §10.10(reject_all/apply partiallyの列挙のみで正史への扱いは無記載)、§10.11(ログ・リプレイの保存対象一覧にも reject 時の扱いは無記載)、§17.1 を確認した。`add-cli-and-sample` の `export-replay` capability の spec.md にも reject_all 起因のターンを除外する規定は無い(確認済み)。この論点は state diff の適用可否(本 change が正本)と、リプレイ生成時に何を「正史」とみなすか(export-replay が正本)の両方にまたがるため、本 change 単独では確定できない。
- **検討した選択肢**: A) reject_all されたターンの narration/events も他ターンと同様に `export replay` の対象に含める(「起きたが世界には影響しなかった一幕」として扱う)。 B) reject_all されたターンは turn_status を applied にしつつも、`review.yaml` の `resulting_turn_status`/`decision` を export-replay 側が参照して当該ターンをスキップする(narration は artifact には残るが正史からは除外)。 C) reject_all は「narration も含めて完全に無かったことにする」ため、実質的に rerun_turn と同じ扱いにし、reject_all という決定自体を廃止して rerun_turn に統合する。
- **推奨案**: B。理由: D107 的な「artifact は削除しない・ログは握り潰さない」という監査可能性の原則を維持しつつ(narration.md 自体は保持)、読者向け正史とは切り離せる。ただし export-replay 側の実装判断が必要。
- **不足インプット**: reject_all されたターンの narration を「読者向け正史」に含めるか除外するかをユーザーに確認する必要がある。確定後、本 change の review.yaml Requirement と `add-cli-and-sample` の export-replay Requirement の双方に反映が必要(cross-change)。
- **Status**: Resolved — D120: artifact保持・export-replayが review.yaml を参照して正史から除外 (openspec/changes/add-session-autonomy/, openspec/changes/add-cli-and-sample/)
